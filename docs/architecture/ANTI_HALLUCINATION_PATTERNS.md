# Anti-Hallucination Patterns in Cronix

This document centralizes the architectural philosophy and defensive mechanisms implemented in the Cronix AI Agent ecosystem (Voice and WhatsApp).

The fundamental principle of the system is: **"Transactional precision kills creativity"**. Large Language Models (LLMs) are excellent at understanding natural intent but inherently poor at performing mathematical calculations, reasoning relative dates, or confirming transactions without hallucinating parameters.

To guarantee `100%` deterministic operations and prevent harmful hallucinations (e.g., booking at incorrect times or inventing clients), Cronix utilizes a **5-Pillar Defensive Architecture**.

---

## 1. Template-Based Response (Response Bypass)

LLMs often "hallucinate" details when asked to rewrite and confirm a transaction that just occurred. To avoid this, we remove the responsibility of the final response from them.

### Implementation (e.g., WhatsApp Agent)
When the execution engine (`RealToolExecutor`) confirms that an appointment was successfully saved in the database:
1. **The LLM loop is aborted:** The result is not sent back to Llama to draft a confirmation.
2. **Strict Template:** The flow is intercepted and a deterministic function is called (`renderBookingSuccessTemplate` in `prompt-builder.ts`).
3. **Result:** The user receives a strict message generated by code:
   > *"Appointment confirmed! We expect you on Thursday, April 12 at 3:00 PM."*

**Benefit:** Mathematical impossibility for the LLM to make a mistake in the confirmed time, while also reducing cost and latency (1 less LLM call).

---

## 2. Input Bypass (Fast Paths)

For exclusive **deterministic read** operations, calling an LLM is unnecessarily slow and error-prone.

### Implementation (e.g., Voice Agent)
In `voice-worker/agent.ts`, the user's intent is evaluated **before** building the prompt:
- If the transcription matches a clear read pattern (e.g., Regex: `/(what appointments do I have today|look for client Maria)/i`), a **Fast Path** is triggered.
- Typescript code executes the query in Supabase immediately.
- Formats the response as text and sends it directly to Deepgram's TTS (Text-to-Speech).
- **Bypasses Groq/Gemini entirely.**

**Benefit:** Latency < 0.5s for schedule queries and zero risk of data hallucination.

---

## 3. Transactional RAG (Grounding vs Long-Term Memory)

Unlike a general knowledge chatbot, a transactional agent needs **immediate precision**, not a "Long-Term Memory" of past conversations stored in vector databases.

### Implementation (Context Injection)
Instead of searching the `ai_memories` table using embeddings (which adds > 500ms of latency), we use **Direct Grounding**:
- In `context-fetcher.ts`, we actively extract the inventory from the database: *Service catalog, Booked slots for the next 14 days, and specific business rules (`aiRules`)*.
- This is dynamically injected as raw text into the System Prompt of every request.

**Benefit:** The LLM has perfect "RAM memory" of what is available at this exact second, eliminating "availability hallucination" (offering already booked slots).

---

## 4. Context Audit

"Context Injection" can become a double-edged sword if we inject too much irrelevant information, contaminating the model's attention (Lost in the middle).

### Implementation
When loading the day's appointments to inject them into the prompt, the system audits and filters the information:
- **Past Cleanup:** If a business owner asks for their schedule at 5:00 PM, the query `.gte('start_at', ...)` is calculated from `Now() - 2 hours`.
- This automatically filters out all morning appointments, which are no longer relevant for operating in the afternoon.

**Benefit:** Keeps the prompt extremely lightweight (saving tokens) and forces the LLM to focus its attention window only on upcoming events.

---

## 5. Date Guards and Strict Prompting (Negative Constraints)

LLMs frequently fail when calculating date jumps (e.g., "the day after tomorrow").

### Date Guards (Argument Interceptors)
If the user says "the day after tomorrow" in the audio, the LLM might try to guess the date in the `get_available_slots` tool.
- We use `detectTemporalIntent` (in `agent.ts`) to intercept the tool call.
- TypeScript deterministically calculates the exact ISO date for "the day after tomorrow" and overrides the LLM's argument **before** sending it to the database.

### Negative Constraints (Prompt Engineering)
In the `prompt.ts` files, military-grade restriction directives ("Negative Constraints") are applied:
- *"If you did not use the tool to search for the client, THEN YOU DO NOT KNOW IF THEY EXIST."*
- *"Under NO circumstances invent schedules. Only offer the schedules returned by the database."*

**Benefit:** Acts as the last line of defense. If the LLM does not have the information verified by a code tool, it is forced to apologize and ask for clarification, rather than inventing an answer to please the user.
