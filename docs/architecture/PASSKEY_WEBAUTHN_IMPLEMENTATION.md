# 🔑 Passkey & WebAuthn Technical Implementation

Cronix implements **Passwordless Authentication** using the **WebAuthn (FIDO2)** standard. This allows business owners to register their devices (iPhone FaceID, Android Fingerprint, YubiKey) as secure authenticators, eliminating the need for traditional passwords.

---

## 🏗️ Technical Stack
-   **Library:** `@simplewebauthn/server` & `@simplewebauthn/browser`.
-   **Database:** Supabase (PostgreSQL) storing public keys and sign counts.
-   **Protocol:** WebAuthn L3 / CTAP2.

---

## 🔄 The WebAuthn Ceremony

Authentication with Passkeys is a two-step process involving a cryptographic "challenge" from the server that the device must "solve".

### 1. Registration Flow
Used to link a new device to a user account.
1.  **Options Request:** The server generates a random **challenge** and a set of `registrationOptions` (RP ID, User ID, Exclude Credentials).
2.  **Challenge Persistence:** The challenge is stored in the `passkey_challenges` table to prevent replay attacks.
3.  **Client Creation:** The browser calls `navigator.credentials.create()`. The user performs biometry.
4.  **Verification:** The server receives the attestation, verifies the cryptographic signature, and stores the **Public Key** (`credential_public_key`) and `credential_id` in the `user_passkeys` table.

### 2. Authentication Flow
Used for subsequent logins.
1.  **Assertion Options:** The server generates a login challenge.
2.  **Sign Ceremony:** The browser calls `navigator.credentials.get()`. The device signs the challenge with its **Private Key**.
3.  **Signature Validation:** The server fetches the stored **Public Key**, verifies the signature, and checks the `sign_count` (to detect cloned authenticators).

## 🛡️ Security Features

### Resident Keys (Discoverable Credentials)
The implementation requires `residentKey: 'required'`, allowing "usernameless" login where the browser can suggest which account to use based on the saved passkey.

### User Verification
Enforced with `userVerification: 'required'`. This ensures that a local biometric check (FaceID/PIN) **must** happen on the device before a signature is generated.

### Anti-Replay Mechanism
Every challenge generated by the `options` endpoint is strictly single-use. Once a verification attempt (registration or login) is made, the challenge is immediately purged from the database.

---

## 📂 Database Schema
-   **`user_passkeys`**: Stores 64-byte public keys, credential IDs, and the last known signature count.
-   **`passkey_challenges`**: Temporary storage for the active cryptographic challenges.
