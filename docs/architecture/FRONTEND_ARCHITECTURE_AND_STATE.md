# 🏗️ Frontend Architecture & State Management

Cronix is built with a highly decoupled architecture that separates the UI layer from the data access logic. This document outlines the patterns used to ensure type-safety, testability, and performance.

---

## 🏛️ The Repository Pattern

The project avoids making direct Supabase calls inside React components. Instead, all data interactions are abstracted into a **Repository Layer** (`lib/repositories/`).

### Why use Repositories?
1.  **Decoupling:** The UI doesn't know the specifics of the Supabase SQL schema. If the database structure changes, only the repository needs an update.
2.  **Shared Logic:** Repositories are designed to accept *any* Supabase client (Server Client or Browser Client). This allows the same logic to be used in **Server Actions**, **API Routes**, and **Client Components**.
3.  **Typed Contracts:** Every repository function returns strictly typed interfaces (e.g., `AppointmentWithRelations`), ensuring that the UI always receives predictable data structures.

### Example: Appointment Repository
```typescript
// lib/repositories/appointments.repo.ts
export async function getDayAppointments(
  supabase: SupabaseClient,
  businessId: string,
  dateStr: string
): Promise<AppointmentWithRelations[]> {
  const { data, error } = await supabase
    .from('appointments')
    .select('id, start_at, status, client:clients(name), service:services(name)')
    .eq('business_id', businessId)
    .gte('start_at', `${dateStr}T00:00:00`)
  // ... handling and casting
}
```

## ⚡ State Management with TanStack Query

For server state (data fetched from the DB), Cronix uses **TanStack Query v5** (React Query). 

### Key Benefits:
-   **Aggressive Caching:** Calendar views and client lists are cached to provide instant navigation.
-   **Automatic Invalidation:** When an appointment is created via a Server Action or a mutation, the specific cache keys (e.g., `['appointments', businessId, month]`) are invalidated to trigger a background refresh.
-   **Optimistic Updates:** Critical UI actions (like marking an appointment as "Attended") use optimistic updates to provide a zero-latency experience.

## 🛡️ Validation & Type Safety

### Cross-Layer Zod Schemas
Validation is centralized in `lib/validations/`. These **Zod schemas** perform triple duty:
1.  **Form Validation:** Used by `react-hook-form` and `@hookform/resolvers/zod` on the client.
2.  **Server-Side Validation:** Used in **Server Actions** to sanitize incoming data before it touches the repositories.
3.  **Type Inference:** TypeScript types are derived directly from the Zod schemas using `z.infer<typeof schema>`, ensuring the types always match the validation rules.

## 🎨 UI System & Shadcn/UI
The presentation layer is built on **Tailwind CSS** and **shadcn/ui**.
-   **Atomic Design:** Components are broken down into small, reusable primitives (Button, Dropdown, Dialog).
-   **Responsive Priority:** Every view is mobile-first, ensuring that business owners can manage their agenda efficiently from a smart phone.
