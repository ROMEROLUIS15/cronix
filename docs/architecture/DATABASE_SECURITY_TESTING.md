# 🛡️ Database Security & RLS Testing Strategy

Cronix follows a **Defense in Depth** approach where security is enforced at the closest possible layer to the data: the database itself. This document details the implementation and verification of **PostgreSQL Row Level Security (RLS)** using **pgTAP** integration tests.

---

## 🏗️ Multi-Tenant Isolation Architecture

The core of Cronix's security is the `business_id` column present in all tenant-scoped tables. Data isolation is not just an application-level concern (where a developer could forget a `.where('business_id', id)` clause); it is a hard constraint enforced by the PostgreSQL engine.

### RLS Implementation Summary
- **Tenant Scope:** Every query executed by a standard user is automatically filtered by `auth.uid()` and its associated `business_id`.
- **Policy Enforcement:** Different policies for `SELECT`, `INSERT`, `UPDATE`, and `DELETE` on tables like `users`, `businesses`, `appointments`, `clients`, etc.
- **Service Role Bypass:** Only the `service_role` (used by Edge Functions and internal scripts) can bypass RLS, provided it explicitly filters by the correct ID.

## 🧪 pgTAP Integration Testing

To ensure that no regression ever breaks our data isolation, we maintain a suite of **45+ automated tests** written in **pgTAP** (a unit testing framework for PostgreSQL).

### Key Test Categories

#### 1. RLS Enablement Verification
Ensures that RLS is not accidentally disabled on critical tables during a migration.
```sql
SELECT ok(
  (SELECT relrowsecurity FROM pg_class WHERE relname = 'appointments' AND relnamespace = 'public'::regnamespace),
  'RLS enabled on appointments'
);
```

#### 2. Anonymous Access Control
Verifies that unauthenticated requests (role `anon`) are strictly rejected from sensitive tables.
```sql
SET ROLE anon;
SELECT throws_ok(
  $q$ INSERT INTO public.businesses (name, owner_id) VALUES ('Evil Corp', gen_random_uuid()) $q$,
  '42501', -- Insufficient Privilege error code
  NULL,
  'anon cannot INSERT into businesses'
);
```

#### 3. Tenant Cross-View Prevention
The most critical test: ensuring **Owner A** cannot see data from **Owner B**.
```sql
SET LOCAL ROLE authenticated;
SET LOCAL "request.jwt.claims" TO '{"sub":"USER_A_UUID","role":"authenticated"}';

SELECT is(
  (SELECT COUNT(*)::INT FROM public.businesses WHERE id = 'BUSINESS_B_UUID'),
  0,
  'Owner A cannot select Owner B business'
);
```

#### 4. Stealth Updates Protection
Ensures that an `UPDATE` attempt on a row that doesn't belong to the user is silently ignored (filters 0 rows) rather than leaking the existence of the row.
```sql
UPDATE public.users SET name = 'Hacked' WHERE id = 'OWNER_B_UUID';
-- Verify that service_role still sees the original name
SET LOCAL ROLE service_role;
SELECT is((SELECT name FROM public.users WHERE id = 'OWNER_B_UUID'), 'Original Name');
```

#### 5. Financial Isolation & AI Privacy
Cronix takes privacy a step further by ensuring even "system" tables are isolated.
- **Financial Blindness:** Owner A cannot see Expenses or Transactions of Business B.
- **Audit Logs:** Metadata and full AI interaction logs are strictly invisible to the `authenticated` role; only the `service_role` can access them for support and observability.
- **Passkey Security:** Biometric credentials (`user_passkeys`) are filtered by `user_id` at the database level.

## 📖 Why pgTAP?

1.  **Atomic Transactions:** Every test runs inside a `BEGIN / ROLLBACK` block. No test data ever pollutes the real database.
2.  **Near-Zero Latency:** Tests run directly inside the engine, making the feedback loop extremely fast.
3.  **Real Role Simulation:** We can toggle between `anon`, `authenticated`, and `service_role` roles within a single script to simulate real attack vectors.
